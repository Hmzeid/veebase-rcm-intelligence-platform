Executive Summary

Autonomous where it is safe, supervised where it matters

Veebase advances each claim through a nine-stage revenue cycle using a pure, deterministic engine that computes a readiness score and a denial-risk score and applies a per-payer rule book for NHIA, private payers, and self-pay. Automation runs only through AUTO gates and stops at every human REVIEW / APPROVE gate, so prohibited actions — auto-accepting low-confidence coding, auto write-offs, submitting a dirty claim — are never executed without a person. The result is faster, cleaner first-pass processing with a complete, immutable audit trail, integrated to payers and TPAs over native JSON, HL7 FHIR R4, and HMAC-signed webhooks.

Use Cases

Where Veebase delivers

Six representative deployments across a provider's revenue cycle. Each maps directly to capabilities that exist in the platform today.

1

Autonomous first-pass processing & exceptions worklist

Hospital billing department

Scenario: High daily claim volume across NHIA and private payers, where most clean claims should move on their own and only true exceptions reach staff.

How Veebase helps: The service layer runs processToGate, looping the engine through every AUTO step (eligibility → prior-auth → charge capture → coding → scrub → submit) and halting the instant a step returns REVIEW or APPROVE. Held claims surface as escalations on a 5-level ladder; everything else advances untouched.

Staff focus on the exceptions worklist instead of touching every claim; clean claims reach submission without manual handling, and each step is logged to the immutable audit trail.
2

Insurer / TPA integration via FHIR + signed webhooks

Payer & TPA connectivity team

Scenario: A private TPA or NHIA gateway needs to push claims in and receive lifecycle status back in real time.

How Veebase helps: Inbound claims arrive as native JSON, batches, or HL7 FHIR R4 Claim resources on the versioned /api/v1 API (API-key auth, scopes read/write/admin). Outbound, subscribers register webhooks and receive HMAC-SHA256-signed events (claim.submitted, claim.denied, claim.paid, escalation.created …) with delivery retries.

A clean two-way contract: FHIR R4 in, signed webhooks out, with idempotency keys making retries safe across instances.
3

Denial management & appeals

Denial & appeals specialists

Scenario: Denied claims need rapid triage — appeal the recoverable ones, write off the rest, but never silently.

How Veebase helps: The DenialManagement agent classifies the denial and selects an appeal strategy. Within the payer's appeal window and above the minimum threshold it drafts an appeal and routes the claim back to SUBMITTED behind an APPROVE gate; otherwise it proposes a write-off — which always requires explicit human approval and raises a Level-3 escalation.

Recoverable denials get appealed with an overturn-probability estimate; write-offs are never auto-executed — they are a governed, approved action.
4

Point-of-registration eligibility pre-check

Front-desk & registration

Scenario: Catch coverage and prior-auth problems before service, not after a denial.

How Veebase helps: POST /api/v1/eligibility runs a real-time eligibility and risk pre-check with no persistence — applying the payer rule book (e.g. prior auth required when amount ≥ the payer threshold: 5,000 EGP NHIA, 3,000 EGP private) and returning a readiness and denial-risk read instantly.

Coverage gaps and missing prior auth are caught at registration, cutting preventable downstream denials.
5

PDF claim ingestion from mixed hospital templates

Intake & ingestion hub

Scenario: Claims arrive as PDFs in many different hospital layouts.

How Veebase helps: POST /api/ingest performs VLM-assisted extraction (multipart file + template) through the pluggable AI router, with a deterministic template-based fallback so extraction never hard-fails. Each response reports the provider and model that produced it.

Mixed-template PDFs become structured claims that enter the same governed pipeline — no fixed input format required.
6

Privacy-sensitive deployment — local AI + field encryption

Compliance & security

Scenario: PHI must not leave the premises, and AI must run without external API calls.

How Veebase helps: Select the local Ollama AI provider (RCM_AI_PROVIDER=ollama) so chat and PDF extraction run on-prem with no external calls, and set RCM_ENCRYPTION_KEY to encrypt sensitive fields (national IDs) at rest with AES-256-GCM. A redaction helper masks 14-digit IDs in logs.

A fully on-prem, air-gapped-capable deployment with encryption-at-rest and an always-available deterministic fallback.

Adoption

How value is realized over the 3-phase rollout

Value compounds as Human-in-the-Loop gates are progressively relaxed from APPROVE toward AUTO — only after each phase has been validated against the audit trail.

Phase 1

Assistive

Agents propose; humans decide. Most gates sit at REVIEW/APPROVE. The platform builds an evidence base of recommendations and outcomes while staff stay fully in control.

Early wins: cleaner first-pass, fewer preventable denials.
Phase 2

Supervised

Proven, low-risk steps shift to AUTO while sensitive actions (coding sign-off, write-offs, high-value > 50,000 EGP) remain gated. Automation share rises; exceptions worklist shrinks.

Compounding: labor efficiency & AR-days benefits scale up.
Phase 3

Scaled

The bulk of routine claims flow autonomously end-to-end; humans own exceptions and governance. Prohibited actions remain permanently human-gated by policy.

Full model value with audit-grade governance intact.
