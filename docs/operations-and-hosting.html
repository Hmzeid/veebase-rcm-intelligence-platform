01Deployment

Deployment topologies

The app is a Next.js 16 standalone build served by server.js under Bun or Node. Persistence, AI providers, the shared store, and encryption are all configured per deployment without code changes — so the same artifact runs in all three topologies below.

a. Single-node

Pilot / Dev
  • One process, SQLite file DB, in-memory rate limiting.
  • Zero external dependencies — the ./scripts/setup.sh default.
  • Ideal for pilots, demos, and development on port 3000.

b. Scaled

Horizontal · HA
  • Multiple stateless app instances behind a load balancer.
  • Managed Postgres (via switch-db.sh) as the shared database.
  • Redis (REDIS_URL) for cross-instance rate limiting; idempotency keys are already DB-shared.

c. Air-gapped / on-prem

No external calls
  • Local Ollama AI provider — chat & PDF extraction run on-prem, no external calls.
  • Field-level encryption at rest (RCM_ENCRYPTION_KEY, AES-256-GCM).
  • Deterministic knowledge-base fallback keeps AI features available offline.
02Lifecycle

Build & run

Prerequisite: Bun (Node also works for the standalone server). The dev server and standalone server both listen on port 3000.

Fresh clone (one command)

# copies .env, installs, generates client, migrates, seeds
./scripts/setup.sh
bun run dev          # http://localhost:3000

Fresh clone (step by step)

cp .env.example .env
bun install
bunx prisma generate
bunx prisma migrate deploy   # or: prisma db push (quick dev)
bun run prisma/seed.ts       # 12 agents, 25 claims, 12 KPIs, 5 RBAC users
bun run dev

Production build & serve

# next build → standalone output (.next/standalone)
bun run build
# serve the standalone server (port 3000)
bun run start        # bun .next/standalone/server.js
Standalone server. bun run build produces a self-contained .next/standalone/server.js with static assets copied in. bun run start (or node .next/standalone/server.js) serves it. Interactive API docs live at /docs; the OpenAPI 3.0 spec at /api/openapi.json.

04Data

Database operations

SQLite is the zero-config default; the Prisma data layer is provider-agnostic and retargets a managed database with no application code changes.

Switch to Postgres

scripts/switch-db.sh postgresql   # or: mysql | sqlite
export DATABASE_URL="postgresql://user:pass@host:5432/rcm"
bunx prisma generate
bunx prisma migrate deploy        # or: db push
bun run prisma/seed.ts            # optional demo data

switch-db.sh rewrites the provider line in prisma/schema.prisma; the rest is standard Prisma.

Migrations & seed

bunx prisma migrate deploy   # apply committed migrations
bun run prisma/seed.ts       # seed demo data

Backups

# SQLite — copy the file (quiesce writes first)
cp db/custom.db db/custom.$(date +%F).db

# Postgres — logical dump
pg_dump "$DATABASE_URL" > rcm-$(date +%F).sql

# Restore
psql "$DATABASE_URL" < rcm-2026-06-13.sql
Runtime DB-path note. A file: SQLite URL is anchored to an absolute path at runtime, so the same DATABASE_URL resolves identically in dev, production, and the standalone build. Override the location with RCM_DATABASE_FILE (absolute path preferred) — useful for mounting the DB on a dedicated volume.
05Operate

Observability & health

Two endpoints cover liveness/readiness and metrics; every API response carries a correlation id, and webhook deliveries are individually logged.

SignalEndpointBehavior
Health GET /api/health Liveness / readiness probe — DB status, claim count, latency. Returns 200 when healthy and 503 when the database is unreachable. Public path (bypasses the auth gate).
Metrics GET /api/metrics Claim totals and by-status, collection rate, pending escalations, webhook delivery failures, user count. JSON by default; Prometheus exposition with ?format=prometheus.
Correlation X-Request-Id Header on all responses — propagate it through logs to trace a single request end-to-end. /api/v1 also returns X-RateLimit-* headers.
Webhook delivery GET /api/v1/webhooks/{id}/deliveries Recent delivery attempts with status, statusCode, and attempts. Send a signed ping via POST /api/v1/webhooks/{id}/test.

Example Prometheus scrape config

scrape_configs:
  - job_name: rcm
    metrics_path: /api/metrics
    params:
      format: [prometheus]
    static_configs:
      - targets: ["rcm.example.com"]

curl -s "https://rcm.example.com/api/metrics?format=prometheus" returns the exposition text; omit the param for JSON.

06Harden

Security checklist for production

The platform ships open by default for demos. Before exposing it, work through this checklist. In production the platform is designed to fail closed — missing or invalid security configuration denies rather than silently allows.

  • Require API auth. Set RCM_REQUIRE_API_AUTH=true so /api/v1 enforces a key even before one is provisioned (disables open bootstrap mode).
  • Sign sessions. Set a long random RCM_SESSION_SECRET and enable the UI gate (RCM_AUTH_ENABLED=true or set RCM_UI_PASSWORD).
  • Encrypt PHI at rest. Set RCM_ENCRYPTION_KEY to AES-256-GCM-encrypt national IDs; the redaction helper masks 14-digit IDs in logs.
  • Restrict CORS. Replace the default * with an explicit RCM_CORS_ORIGINS allow-list of your trusted front-ends/integrators.
  • Provision scoped API keys. Issue least-privilege keys via POST /api/v1/keys (read / write / admin); secrets are shown once and stored only as SHA-256 hashes.
  • Protect metrics. Put /api/metrics behind network controls (internal network / firewall / proxy auth) — it is not key-gated.
  • Terminate TLS at the proxy. Serve over HTTPS; the standalone server should sit behind a TLS-terminating reverse proxy.
  • Keep webhooks egress-safe. Leave RCM_WEBHOOK_ALLOW_PRIVATE=false to block delivery to private addresses (SSRF mitigation).
  • Rotate keys regularly. Rotate API keys, the session secret, and the master key on a schedule and after any suspected exposure.
Fail-closed in production. Security-sensitive misconfiguration (e.g. an invalid master key) is treated as a denial rather than an open bypass when running in production, so a broken setting can never silently widen access.
07Run

Runbook

Day-2 procedures for operators.

▶ Start / stop

Build once, then run the standalone server; stop by terminating the process (or your supervisor/orchestrator).

bun run build && bun run start   # start (port 3000)
# stop: terminate the server.js process / scale the deployment to 0

🔑 Rotate API keys

Provision a replacement key, cut integrators over, then disable the old one. The secret is returned exactly once.

curl -X POST https://rcm.example.com/api/v1/keys \
  -H "X-API-Key: $RCM_MASTER_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "name": "tpa-prod", "scopes": ["read","write"] }'

🔁 Rotate session secret

Update RCM_SESSION_SECRET to a new long random value and restart. Existing UI sessions are invalidated and users re-authenticate.

⤢ Scale out (add Redis)

Move to multiple instances: switch to Postgres, set REDIS_URL for shared rate limiting (install optional ioredis), and run N stateless app instances behind the load balancer. Idempotency keys are already DB-shared, so retries stay safe.

💾 Backup & restore

SQLite: copy db/custom.db (quiesce writes first). Postgres: pg_dump / psql restore. Test restores periodically. See Database operations.

08Scale

Scaling notes

The app scales horizontally because state lives in shared infrastructure, not in the process.

Stateless app instances

Each app instance is stateless — add or remove instances behind the load balancer freely. All durable state lives in the shared database.

Idempotency is DB-shared

Idempotency keys are stored in the primary database, so a retried create request returns the original claim regardless of which instance handles it — even without Redis.

Rate limiting needs Redis at scale

In-memory counters are per-instance. For accurate multi-instance limiting set REDIS_URL; the Redis-backed limiter becomes authoritative while the Edge middleware stays the per-instance first line.

AI provider failover

Chat and PDF extraction walk active → fallbacks → deterministic fallback. If every provider is down the platform still returns a deterministic knowledge-base answer, so AI features never hard-fail.
